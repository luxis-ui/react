## Before You Start

RichTextEditor renders a `contentEditable` div with `role="textbox"` and a companion formatting toolbar.
It relies on the browser's `execCommand` API for formatting and the WAI-ARIA `textbox` (multiline) pattern for accessibility.
The Quick Start example below provides a basic initialization of RichTextEditor.

<Callout type="warning" title="Security: always sanitize user HTML">
  RichTextEditor produces raw HTML. Before persisting or rendering user-generated content
  elsewhere in your app, run it through a sanitizer such as
  [DOMPurify](https://github.com/cure53/DOMPurify) via the `sanitize` prop.
</Callout>

<Callout type="tip" title="New to Luxis UI?">
  Check our Getting Started guide for overarching conventions.
</Callout>

### Controlled

When to use: When a parent component needs to read or drive the content.

<Callout type="info" title="Controlled vs Uncontrolled">
  By default RichTextEditor manages its own content (uncontrolled via `defaultValue`).
  Use the controlled pattern — `value` + `onChange` — when syncing with a form
  library, URL param, or another component.
</Callout>

<!--

### Custom Link Insertion

Override the default `window.prompt` with your own modal using `onInsertLink`. The callback receives an `insert` function — call it with the resolved URL when the user confirms.

<!--
EXAMPLE: Custom Link Insertion
PURPOSE: Replacing the browser prompt with a custom link modal
OUTPUT: RichTextEditor that opens a custom dialog for link URLs
USE WHEN: The default browser prompt is not acceptable UX
SOURCE: Derived from RichTextEditor.tsx onInsertLink prop
-->
```tsx
import { RichTextEditor } from '@luxis-ui/react';

export default function CustomLinkExample() {
  const handleInsertLink = (insert: (url: string) => void) => {
    // Replace with your own modal or URL picker
    openLinkModal().then((url) => {
      if (url) insert(url);
    });
  };

  return (
    <RichTextEditor
      label="Content with Custom Link Dialog"
      onInsertLink={handleInsertLink}
    />
  );
}
```

### With HTML Sanitizer

Pass a `sanitize` function to strip dangerous markup before it is injected via `innerHTML`. This is essential when the `value` originates from user-generated content.

<!--
EXAMPLE: With HTML Sanitizer
PURPOSE: Preventing XSS by sanitizing the value before rendering
OUTPUT: RichTextEditor that sanitizes HTML through DOMPurify
USE WHEN: value prop contains untrusted or user-generated HTML
SOURCE: Derived from RichTextEditor.tsx sanitize prop
-->
```tsx
import DOMPurify from 'dompurify';
import { RichTextEditor } from '@luxis-ui/react';

export default function SanitizedExample() {
  return (
    <RichTextEditor
      label="User Content"
      value={untrustedHtml}
      sanitize={(html) => DOMPurify.sanitize(html)}
      onChange={(h) => handleChange(h)}
    />
  );
}
```

## RichTextEditor Best Practices

<DoAndDont
  good={`// ✅ Always provide a visible label or aria-label
<RichTextEditor label="Article Body" />`}
  bad={`// ❌ Leaving the editor without an accessible name
<RichTextEditor placeholder="Type here..." />`}
/>

<DoAndDont
  good={`// ✅ Sanitize untrusted HTML via the sanitize prop
<RichTextEditor
  value={userHtml}
  sanitize={(html) => DOMPurify.sanitize(html)}
/>`}
  bad={`// ❌ Rendering raw user HTML without sanitization
<RichTextEditor value={userHtml} />`}
/>

<DoAndDont
  good={`// ✅ Use onInsertLink to control the URL input experience
<RichTextEditor onInsertLink={(insert) => openModal().then(insert)} />`}
  bad={`// ❌ Relying on the default browser prompt in production UIs
<RichTextEditor />`}
/>

<DoAndDont
  good={`// ✅ Use design tokens for theming
<RichTextEditor style={{ '--lxs-richtext-border': '2px solid blue' }} />`}
  bad={`// ❌ Overriding internal styles with !important
<RichTextEditor className="force-blue-border" />`}
/>

## RichTextEditor Accessibility Guide

RichTextEditor follows the WAI-ARIA `textbox` (multiline) pattern and meets WCAG AA standards.

### Keyboard Shortcuts

| Key | Action |
|-----|--------|
| <KeyboardKey>Tab</KeyboardKey> | Moves focus into the editor area |
| <KeyboardKey>Shift+Tab</KeyboardKey> | Moves focus out of the editor area |
| <KeyboardKey>Ctrl+B</KeyboardKey> | Bold (browser native via `execCommand`) |
| <KeyboardKey>Ctrl+I</KeyboardKey> | Italic |
| <KeyboardKey>Ctrl+U</KeyboardKey> | Underline |
| <KeyboardKey>Ctrl+Z</KeyboardKey> | Undo |
| <KeyboardKey>Ctrl+Y</KeyboardKey> | Redo |

> Toolbar buttons are excluded from the tab order (`tabIndex=-1`) to keep the focus flow lean. They are clickable with a mouse and can be exposed to keyboard users via your own toolbar navigation if needed.

### ARIA Attributes

| Attribute | Value | When |
|-----------|-------|------|
| `role` | `"textbox"` | Always; establishes the editor as an input widget |
| `aria-multiline` | `"true"` | Always; signals multi-line text entry |
| `aria-label` | string | When no visible `label` prop is provided |
| `aria-invalid` | `"true" \| undefined` | When `error` prop is set |
| `aria-required` | `"true" \| undefined` | When `required` prop is true |
| `aria-describedby` | string | Links to helper text, error, or success message IDs |

### Screen Reader Behavior

When focused, screen readers announce the element as a textbox (multiline), followed by its accessible name (from `label` or `aria-label`). Validation messages (`error`, `success`) are linked via `aria-describedby`; the error paragraph carries `role="alert"` for immediate announcement. The character counter uses `aria-live="polite"` to announce updates without interrupting the user.

### Accessibility Checklist

- [ ] Provide `label` or `aria-label` on every instance
- [ ] Sanitize `value` content with the `sanitize` prop when the source is untrusted
- [ ] Test keyboard navigation — Tab into editor, type, Tab out
- [ ] Confirm error messages are announced by assistive technology
- [ ] Verify focus outline is clearly visible on the editor container

## Troubleshooting RichTextEditor

### Common Mistakes

| Symptom | Wrong ❌ | Right ✅ |
|---------|---------|---------|
| Content not updating in controlled mode | Using `defaultValue` and expecting re-renders to update it | Use `value` + `onChange` for controlled behavior |
| XSS vulnerability | Rendering `value` from user input without sanitization | Pass `sanitize={(html) => DOMPurify.sanitize(html)}` |
| Link insertion shows browser prompt | Not providing `onInsertLink` | Pass a custom `onInsertLink` handler wired to your modal |
| Screen readers silent | No `label` and no `aria-label` | Always provide at least one accessible name |
| Counter not appearing | Setting only `maxLength` without `showCount` | Set `showCount` to `true`, or use `maxLength` alone (counter appears automatically with `maxLength`) |
| Styles not applying | Hardcoding conflicting inline styles | Use CSS variables via `style` prop or `wrapperClassName` |

<Callout type="warning" title="Next.js / SSR">
  RichTextEditor uses `contentEditable` and browser APIs (`document.execCommand`,
  `window.prompt`). Mark any component that renders it with `"use client"` in
  Next.js App Router to avoid server-side rendering errors.
</Callout>

### Frequently Asked Questions

**Q: Can I use RichTextEditor with Zod or Yup validation?**
A: Yes. Use the controlled pattern (`value` + `onChange`) and pass the validation error string to the `error` prop. The component renders it in the error slot and applies `aria-invalid` automatically.

**Q: Why does the toolbar use `execCommand` instead of a modern API?**
A: `document.execCommand` is the only cross-browser API for `contentEditable` formatting. The component wraps it safely and a `NOSONAR` comment acknowledges the deprecation. No robust, broadly supported replacement exists for this use case today.

**Q: How do I persist the HTML content to a database?**
A: Read the `html` argument passed to `onChange`. Sanitize it with DOMPurify before storing, and set `sanitize={(html) => DOMPurify.sanitize(html)}` before rendering stored HTML back into the editor.

**Q: Can I extend the toolbar with custom buttons?**
A: The `toolbarButtons` prop only accepts the predefined `ToolbarButton` keys. For fully custom toolbar actions, wrap RichTextEditor with `hideToolbar` and build your own toolbar that calls methods on the forwarded ref.

**Q: Does RichTextEditor work inside a `<dialog>` or modal?**
A: Yes. Focus management is handled natively. Ensure your modal traps focus correctly so Tab cycles within the modal, and the editor will behave as expected.
